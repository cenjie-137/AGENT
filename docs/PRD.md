# B端智能售前方案生成系统 — 产品需求文档 (PRD)

## 1. 项目概述

### 1.1 背景
B端产品售前阶段存在大量重复性工作：理解客户需求、匹配产品能力、分析竞品、撰写方案。这些工作依赖资深售前/产品经理的个人经验，新人上手慢、产出质量不稳定。

本项目旨在构建一套**基于大模型+RAG的通用售前方案生成框架**，将售前工作中的结构化部分（需求解析、产品匹配、竞品对比、方案组装）自动化，让一线销售/售前人员能在分钟级产出高质量的初步方案。

### 1.2 目标
- **短期**：在科大讯飞教育产品线验证核心流程，辅助完成真实售前场景
- **中期**：抽象为通用框架，支持任意B端行业的产品方案生成
- **长期**：开源为可复用的AI售前工具，降低B端产品售前门槛

### 1.3 核心价值主张
> "将B端售前方案产出时间从2天缩短至10分钟，同时保证方案的专业性和一致性"

## 2. 目标用户

| 角色 | 痛点 | 使用场景 |
|------|------|----------|
| **售前工程师** | 重复写方案、产品知识记不住、竞品信息查不全 | 收到客户需求后快速生成初步方案 |
| **产品经理** | 需求分析耗时、竞品跟踪不及时 | 辅助进行市场研究和竞品分析 |
| **销售代表** | 技术能力弱、无法独立应对技术型客户 | 拿到客户会议纪要后自动生成应对方案 |
| **行业新人** | 不熟悉产品线、不了解行业最佳实践 | 通过系统快速学习和产出 |

## 3. 核心功能模块

### 3.1 需求智能解析（Intent Parser）
**输入**：客户需求描述（自然语言、会议纪要、问卷等）
**输出**：
- 结构化需求清单（功能需求 / 非功能需求）
- 客户画像（行业、规模、决策链角色）
- 隐性需求推断（基于行业Know-How）
- 需求优先级建议

### 3.2 产品能力匹配（Product Matcher）
**输入**：结构化需求 + 产品知识库
**输出**：
- 产品-需求匹配矩阵（匹配度评分：高/中/低）
- 标准功能覆盖清单
- 需要定制开发的Gap分析
- 推荐的产品组合方案

### 3.3 竞品智能分析（Competitor Analyst）
**输入**：客户行业和需求关键词
**输出**：
- 主要竞品识别（Top 3-5）
- 多维度对比表（功能、价格、服务、生态）
- 我方差异化优势提炼
- 投标策略建议

### 3.4 方案智能生成（Proposal Generator）
**输入**：需求解析结果 + 产品匹配结果 + 竞品分析结果
**输出**：
- 方案大纲（含技术架构建议）
- 实施路径与里程碑
- 资源投入估算
- 风险与应对策略
- 可导出为 Markdown / PDF / PPT

## 6. 技术方案

### 6.1 技术栈选型

| 模块 | 技术选型 | 理由 |
|------|---------|------|
| 前端界面 | Gradio / Streamlit | 快速搭建Demo，适合开源展示 |
| API服务 | FastAPI | Python生态，异步高性能 |
| LLM调用 | OpenAI API / 讯飞星火API | 多模型兼容，可切换 |
| RAG框架 | LangChain + ChromaDB | 主流方案，社区活跃 |
| 向量数据库 | ChromaDB / FAISS | 轻量，本地可运行 |
| 文档解析 | unstructured + PyPDF2 | 支持多格式 |
| 报告生成 | Markdown + WeasyPrint | 轻量PDF导出 |
| 工作流编排 | LangGraph | 支持复杂多步推理 |

### 6.2 知识库设计（模块化、可替换）

```
data/
├── knowledge_base/
│   ├── education/              # 教育行业（当前验证）
│   │   ├── products/           # 产品文档
│   │   ├── cases/              # 客户案例
│   │   └── policies/           # 行业政策
│   ├── healthcare/             # 医疗行业（预留）
│   └── manufacturing/          # 工业行业（预留）
├── competitor_db/
│   └── education_competitors.json
└── templates/
    ├── proposal_template.md
    └── comparison_table_template.md
```

### 6.3 Prompt工程策略

采用**分层Prompt架构**：
- **系统级Prompt**：定义Agent角色和专业领域
- **任务级Prompt**：每个模块的独立任务定义
- **示例级Prompt**：Few-shot示例提升输出质量
- **校验级Prompt**：输出格式检查和自我修正

## 7. 行业适配设计

### 7.1 适配原则
- **知识库可插拔**：替换data/knowledge_base/下的内容即可服务新行业
- **Prompt模板可配置**：行业特定术语和分析维度通过配置文件注入
- **竞品源可扩展**：支持从数据库、网络搜索、用户上传多种来源获取竞品信息

### 7.2 当前验证行业：教育信息化
- 产品库：讯飞教育产品线（智慧课堂、考试评测、区域治理等）
- 竞品库：腾讯教育、阿里钉钉教育、希沃、好未来
- 政策库：教育部AI+教育相关政策

### 7.3 预留扩展行业
- 医疗信息化（HIS/PACS/智慧医院）
- 工业智能制造（MES/工业互联网）
- 金融科技（风控/数据中台）

## 9. 成功指标

- **功能指标**：覆盖售前方案生成的核心环节（需求→匹配→竞品→方案）
- **效率指标**：方案产出时间 < 10分钟（对比人工2天）
- **质量指标**：方案专业度评分 >= 4/5（由售前专家评估）
- **通用性指标**：支持至少3个行业的知识库替换
- **开源指标**：GitHub Stars >= 100，README完整度 >= 90%

## 10. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| LLM输出不稳定 | 方案质量波动 | 增加输出校验层 + Few-shot示例 |
| 知识库构建耗时 | 项目进度延迟 | 先用公开资料构建MVP，再逐步完善 |
| 竞品信息获取难 | 竞品分析不准确 | 结合公开资料+网络搜索，明确信息时效性 |
| 行业差异大 | 通用框架难以覆盖 | 采用插件化架构，核心流程固定，行业内容可替换 |
